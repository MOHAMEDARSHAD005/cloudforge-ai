# OBSERVABILITY.md
> Observability strategy for CloudForge AI: structured logging, metrics, tracing, dashboards, and alerts.
> Every system component must meet the standards defined here.
> Last updated: June 2026

---

## Principles

1. **Structured JSON only.** No `console.log()`, no `print()`, no unstructured text in production logs.
2. **TraceId on everything.** Every request, job, agent execution, and artifact write carries a `traceId`.
3. **Failure is always visible.** Every error emits a log at `level: "error"`, increments a failure counter metric, and writes a `JobEvent` record.
4. **Cost is always visible.** Every LLM call records token usage. Total job cost is always computable.
5. **Dashboards reflect SLOs.** Every SLO target from `project-context.md` has a corresponding dashboard panel and alert.

## Structured Logging

### Log Format (All Services)

Every log line is a single JSON object:

```json
{
  "timestamp": "2026-06-24T10:00:00.000Z",
  "level": "info",
  "service": "api-nest",
  "version": "1.2.3",
  "environment": "production",
  "traceId": "trace_abc123",
  "jobId": "job_xyz",
  "projectId": "proj_abc",
  "userId": "usr_def",
  "message": "Agent architecture complete",
  "context": {
    "agent": "architecture",
    "durationMs": 8420,
    "inputTokens": 1850,
    "outputTokens": 2240
  }
}
```

### Log Levels

| Level | When to use |
|---|---|
| `debug` | Detailed internal state (disabled in production by default) |
| `info` | Normal operations: request received, agent started, agent complete, job complete |
| `warn` | Degraded but not broken: retry attempt, non-fatal agent failure, token budget warning |
| `error` | Failure state: agent failed after retries, validation error, infrastructure error |

### Required Context Fields

| Field | Present on | Notes |
|---|---|---|
| `traceId` | Every log line | Generated by NestJS middleware, propagated to FastAPI |
| `jobId` | Job-related logs | Null for non-job requests |
| `projectId` | Job-related logs | Null for non-job requests |
| `userId` | Authenticated request logs | Null for unauthenticated |
| `agent` | Agent execution logs | e.g., `"planner"`, `"reviewer"` |
| `durationMs` | Completion logs | Always include on agent:complete, job:complete |

### Implementation

**NestJS (Winston):**
```typescript
// libs/logger/logger.service.ts
import winston from 'winston';

const logger = winston.createLogger({
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()
  ),
  transports: [new winston.transports.Console()],
});

// Middleware adds traceId to every request
// LoggerService binds traceId to AsyncLocalStorage context
// All subsequent logger.info() calls include traceId automatically
```

**FastAPI (structlog):**
```python
# app/core/logging.py
import structlog

structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ]
)

# In request handler:
structlog.contextvars.bind_contextvars(
    trace_id=request.headers.get("X-Trace-Id"),
    job_id=body.job_id,
)
# All subsequent logger.info() calls include trace_id and job_id
```

### CI Enforcement

- **NestJS:** ESLint rule `no-console` set to `"error"` in `.eslintrc`
- **FastAPI:** flake8 plugin `flake8-print` set to fail on bare `print()` calls
- **Pre-commit hook:** `grep -r "console.log\|print(" apps/` fails if found

## Metrics

### Metric Categories

**Agent Metrics:**

| Metric Name | Type | Labels | Description |
|---|---|---|---|
| `cf_agent_duration_ms` | Histogram | `agent`, `status` | LLM call duration per agent |
| `cf_agent_input_tokens_total` | Counter | `agent`, `model_name` | Total input tokens consumed |
| `cf_agent_output_tokens_total` | Counter | `agent`, `model_name` | Total output tokens produced |
| `cf_agent_failure_total` | Counter | `agent`, `failure_reason` | Failed agent executions |
| `cf_agent_retry_total` | Counter | `agent` | Agent retry attempts |
| `cf_agent_cost_usd_total` | Counter | `agent`, `model_name` | Estimated LLM cost in USD |

**Job Metrics:**

| Metric Name | Type | Labels | Description |
|---|---|---|---|
| `cf_job_duration_ms` | Histogram | `status` | Total job wall-clock time |
| `cf_job_created_total` | Counter | — | Jobs submitted |
| `cf_job_complete_total` | Counter | `status` | Jobs completed by status (COMPLETE, PARTIAL, FAILED) |
| `cf_job_artifact_count` | Histogram | `status` | Artifacts produced per job |

**Queue Metrics:**

| Metric Name | Type | Labels | Description |
|---|---|---|---|
| `cf_queue_depth` | Gauge | `queue_name` | Current jobs waiting in queue |
| `cf_queue_wait_ms` | Histogram | — | Time from enqueue to worker pickup |
| `cf_queue_dlq_total` | Counter | — | Jobs moved to dead letter queue |

**Infrastructure Metrics (from ECS/CloudWatch):**

| Metric | Source |
|---|---|
| CPU utilization per service | ECS CloudWatch built-in |
| Memory utilization per service | ECS CloudWatch built-in |
| ALB request count | ALB CloudWatch built-in |
| ALB 5xx error rate | ALB CloudWatch built-in |
| RDS connection count | RDS CloudWatch built-in |
| RDS query latency | RDS CloudWatch built-in |
| ElastiCache cache hit ratio | ElastiCache CloudWatch built-in |
| ElastiCache memory usage | ElastiCache CloudWatch built-in |

### Metric Collection

**Phase 1–4 (Pre-production):** Metrics are written to structured logs. CloudWatch Log Insights queries extract metrics from JSON log fields.

**Phase 5 (Production):** Prometheus-format metrics endpoint (`/metrics`) on NestJS and FastAPI. CloudWatch Agent scrapes and forwards to CloudWatch. Optional: Grafana Cloud or Datadog for richer dashboards (per OQ-005 resolution).

## Distributed Tracing

### Current Approach: Manual traceId Propagation (Phase 1–4)

CloudForge AI uses a manually propagated `traceId` rather than full OpenTelemetry distributed tracing. This achieves ~90% of the operational value with significantly less setup complexity.

**Trace flow:**

```
Browser Request
  │  X-Trace-Id: trace_abc123 (generated by NestJS middleware if not present)
  ▼
NestJS
  │  Logs: { traceId: "trace_abc123", message: "Request received" }
  │  Adds X-Trace-Id header to internal requests
  ▼
BullMQ Job Payload
  │  { traceId: "trace_abc123", jobId: "job_xyz", ... }
  ▼
NestJS Worker
  │  Logs: { traceId: "trace_abc123", message: "Job picked up by worker" }
  ▼
FastAPI (POST /generate)
  │  Headers: X-Trace-Id: trace_abc123
  │  structlog.contextvars.bind_contextvars(trace_id="trace_abc123")
  │  All subsequent logs: { trace_id: "trace_abc123" }
  ▼
FastAPI → Anthropic API
  │  Logs: { trace_id: "trace_abc123", message: "LLM call started", agent: "planner" }
  ▼
FastAPI → NestJS Callback
  │  Body: { traceId: "trace_abc123", event: "agent:complete", ... }
  ▼
NestJS → PostgreSQL
  │  Logs: { traceId: "trace_abc123", message: "Artifact persisted" }
  ▼
NestJS → WebSocket
  │  Event: { traceId: "trace_abc123", event: "agent:complete", ... }
```

**To reconstruct a job execution in CloudWatch Logs Insights:**
```sql
fields @timestamp, service, level, message, agent, durationMs
| filter traceId = "trace_abc123"
| sort @timestamp asc
```

### Future Approach: OpenTelemetry (Phase 5+)

When the team grows and Grafana or Datadog is adopted, migrate to OpenTelemetry:
- NestJS: `@opentelemetry/sdk-node` with auto-instrumentation
- FastAPI: `opentelemetry-sdk-python` with `opentelemetry-instrumentation-fastapi`
- Traces exported to Grafana Tempo or Datadog APM

This is deferred — manual `traceId` is sufficient for Phase 1–5.

## Dashboards

| Panel | Metric | Visualization |
|---|---|---|
| Job success rate (24h) | `cf_job_complete_total` by status | Pie chart |

| Active queue depth | `cf_queue_depth` | Gauge |
| ALB 5xx error rate | ALB CloudWatch | Time series |
| NestJS CPU/Memory | ECS CloudWatch | Time series |
| FastAPI CPU/Memory | ECS CloudWatch | Time series |

### Dashboard 2: Agent Performance

Purpose: Per-agent reliability and cost tracking.

| Panel | Metric | Visualization |
|---|---|---|
| Agent duration P50/P95 | `cf_agent_duration_ms` by agent | Grouped bar chart |
| Agent failure rate by agent | `cf_agent_failure_total` / job count | Table |
| Agent retry rate by agent | `cf_agent_retry_total` | Bar chart |
| Token usage by agent | `cf_agent_input_tokens_total` + `cf_agent_output_tokens_total` | Stacked bar |
| Cost per agent (rolling 24h) | `cf_agent_cost_usd_total` | Bar chart |

### Dashboard 3: Queue Health

Purpose: Detect queue saturation, DLQ buildup, and worker bottlenecks.

| Panel | Metric | Visualization |
|---|---|---|
| Queue depth over time | `cf_queue_depth` | Time series |
| Queue wait time P95 | `cf_queue_wait_ms` | Time series |
| Worker throughput (jobs/hour) | `cf_job_complete_total` | Time series |
| DLQ depth | `cf_queue_dlq_total` | Gauge with alert threshold |

### Dashboard 4: Cost & Token Usage

Purpose: LLM cost tracking, budget monitoring, anomaly detection.

| Panel | Metric | Visualization |
|---|---|---|
| Total LLM cost (rolling 24h) | `cf_agent_cost_usd_total` | Stat |
| Cost per job (rolling 24h average) | Derived | Stat |
| Token usage by model (rolling 24h) | `cf_agent_input_tokens_total` by model_name | Time series |
| Cost by agent type | `cf_agent_cost_usd_total` by agent | Pie chart |
| Rolling 1h cost vs baseline | Derived comparison | Time series with anomaly band |

### Dashboard 5: Error Analysis

Purpose: Fast triage of errors by type, agent, and failure reason.

| Panel | Metric | Visualization |
|---|---|---|
| Error rate by service (NestJS / FastAPI) | Log-derived | Time series |

## Alerts

All alerts link to a runbook in `RUNBOOKS.md`.

### Tier 1 — Critical (Page On-Call Immediately)

| Alert | Condition | Runbook |
|---|---|---|
| High job fatal failure rate | Fatal job failure rate > 5% in 5 minutes | RB-001 |
| Redis unavailable | BullMQ connection errors for > 30 seconds | RB-002 |
| PostgreSQL unavailable | DB connection errors for > 30 seconds | RB-003 |
| FastAPI unreachable | NestJS Worker HTTP failures for > 60 seconds | RB-004 |
| DLQ spike | > 10 jobs in DLQ in 5 minutes | RB-005 |
| ALB 5xx rate spike | > 5% of requests returning 5xx for > 2 minutes | RB-006 |

### Tier 2 — Warning (Notify Slack, No Page)

| Alert | Condition | Runbook |
|---|---|---|
| Queue depth high | Queue depth > 100 for > 5 minutes | RB-007 |
| Token usage spike | Per-job token usage > 3× rolling 1h baseline | RB-008 |
| Single agent failure rate high | Any agent failure rate > 10% in 5 minutes | RB-009 |
| Reviewer agent degraded | Reviewer failure rate > 5% in 5 minutes | RB-010 |
| ECS task count below minimum | NestJS or FastAPI task count < min (2) | RB-011 |
| High RDS connection count | RDS connections > 80% of max | RB-012 |
| Pydantic validation failure spike | Validation failures > 3% in 5 minutes | RB-013 |
| Auth failure rate spike | Auth failures > 20 req/min from single IP | RB-014 |

### Alert Configuration (CloudWatch)

```hcl
# CloudWatch alarm — fatal job failure rate
resource "aws_cloudwatch_metric_alarm" "fatal_job_failure_rate" {
  alarm_name          = "cloudforge-fatal-job-failure-rate"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 1
  metric_name         = "cf_job_complete_total_FAILED"
  namespace           = "CloudForge/Jobs"
  period              = 300   # 5 minutes
  statistic           = "Sum"
  threshold           = 5
  alarm_description   = "Fatal job failure rate exceeds 5 in 5 minutes"
  alarm_actions       = [aws_sns_topic.on_call.arn]
  ok_actions          = [aws_sns_topic.on_call.arn]
}
```

## JobEvent Log Schema

Every significant event in a job's lifecycle is persisted as a `JobEvent` record. This creates a queryable audit trail independent of application logs.

```typescript
// JobEvent types
type JobEventType =
  | 'job:created'
  | 'job:started'
  | 'agent:started'
  | 'agent:complete'
  | 'agent:failed'
  | 'agent:retry'
  | 'job:complete'
  | 'job:partial'
  | 'job:failed'
  | 'job:stuck';

// Example JobEvent records for a complete job
[
  { jobId, event: 'job:created',       payload: { userId, prompt } },
  { jobId, event: 'job:started',       payload: { workerId } },
  { jobId, event: 'agent:started',     payload: { agent: 'planner' } },
  { jobId, event: 'agent:complete',    payload: { agent: 'planner', durationMs: 9420, tokens: 1730 } },
  { jobId, event: 'agent:started',     payload: { agent: 'architecture' } },
  { jobId, event: 'agent:started',     payload: { agent: 'aws_expert' } },
  { jobId, event: 'agent:complete',    payload: { agent: 'architecture', durationMs: 11200, tokens: 3800 } },
  { jobId, event: 'agent:complete',    payload: { agent: 'aws_expert', durationMs: 10800, tokens: 3600 } },
  { jobId, event: 'agent:started',     payload: { agent: 'security' } },
  { jobId, event: 'agent:failed',      payload: { agent: 'security', retries: 2, fatal: false } },
  { jobId, event: 'agent:complete',    payload: { agent: 'cost', durationMs: 8900, tokens: 2100 } },
  { jobId, event: 'agent:complete',    payload: { agent: 'diagram', durationMs: 6200, tokens: 1400 } },
  { jobId, event: 'agent:complete',    payload: { agent: 'terraform', durationMs: 22100, tokens: 7200 } },
  { jobId, event: 'agent:complete',    payload: { agent: 'reviewer', durationMs: 14300, tokens: 8900 } },
  { jobId, event: 'job:partial',       payload: { totalDurationMs: 51240, failedAgents: ['security'] } },
]
```

This table is the ground truth for debugging any job. It is always queryable even if logs have rolled over.
